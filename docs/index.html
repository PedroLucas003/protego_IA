// 03 — BACKEND

7 Módulos do Backend

MOD · 01
TratarDados
JSON Schema para validação de todos os payloads MQTT e REST. Garante consistência e integridade dos dados entre os módulos.
MOD · 02
GestaoDevice
Gerencia as câmeras ESP32 cadastradas. Controla status online/offline, IP e ativação dos dispositivos de campo.
MOD · 03
BrokerMQTT
Camada de comunicação em tempo real. Tópicos: reconhecimento/facial, reconhecimento/status, reconhecimento/comando.
MOD · 04
OTA
Over The Air Update — atualização remota do firmware do ESP32 sem necessidade de conexão física via USB.
MOD · 05
ServicosAPI
API REST principal consumida pelo Flutter. Endpoints para pessoas, mandados, histórico criminal e detecções.
MOD · 06
Negocio
Regras de negócio centralizadas: níveis de alerta por perigo, cooldown de notificações, validação de prova de vida.
MOD · 07
Administracao
Painel para gestão de alvos, geração de encodings via foto, logs do sistema e relatórios de abordagens.
// 04 — OPERAÇÃO

Fluxo de Uso

01

Policial pluga a bodycam

O ESP32-CAM inicializa automaticamente, conecta ao WiFi e começa a transmitir o stream MJPEG. Nenhum software precisa ser aberto manualmente.

02

IA processa os rostos

O InsightFace ArcFace detecta e gera embeddings de 512 dimensões para cada rosto no frame. A comparação é feita por similaridade coseno contra o banco de alvos.

03

Alvo identificado

A ficha completa é exibida em overlay: nome, nível de perigo, mandados ativos, crimes e emoção detectada pelo DeepFace. Beep sonoro proporcional ao perigo.

04

Alerta via MQTT → Flutter

A ficha é publicada no broker MQTT, o FastAPI recebe e encaminha via WebSocket para o app Flutter do policial em tempo real.

05

Registro no banco

A abordagem é registrada no PostgreSQL com timestamp, câmera, confiança e foto capturada — formando o histórico completo de operações.

PORT MAPPING — RAILWAY
28372
PORTA EXTERNA (proxy)
→
1883
PORTA INTERNA (EMQX)

No Railway a porta 1883 é a padrão MQTT dentro do container. A porta 28372 é a porta pública gerada pelo Railway que redireciona para ela. Isso é padrão em ambientes cloud/containerizados — a infraestrutura faz o mapeamento de portas automaticamente.

CONCLUSÃO

Foi possível integrar com sucesso o broker MQTT EMQX ao banco de dados PostgreSQL, utilizando uma arquitetura adaptada para ambiente de produção no Railway. O Rule Engine do EMQX intercepta cada mensagem publicada e a encaminha via HTTP para o FastAPI, que persiste os dados e os disponibiliza via API REST — garantindo o funcionamento completo do fluxo de dados de ponta a ponta.

// 10 — EXECUÇÃO

Como Executar o Projeto

01

CLONE O REPOSITÓRIO

git clone https://github.com/PedroLucas003/protego_IA.git
cd protego_IA
02

SUBA A INFRAESTRUTURA COM DOCKER

Inicia os 4 serviços em containers: Mosquitto, EMQX, FastAPI e PostgreSQL.

cd backend_nuvem
docker compose up -d
03

INSTALE AS DEPENDÊNCIAS PYTHON

pip install -r requirements.txt
04

ADICIONE FOTOS AO BANCO DE FACES

Coloque arquivos .jpg ou .png na pasta banco_faces/. O nome do arquivo vira o nome reconhecido.

banco_faces/
  joao_silva.jpg    → reconhecido como JOAO_SILVA
  criminoso_1.jpg  → reconhecido como CRIMINOSO_1
05

CONFIGURE O IP DO ESP32 E EXECUTE A IA

# Em reconhecimento_final.py
IP_ESP32 = "SEU_IP_AQUI"  # ex: 192.168.1.91

python reconhecimento_final.py
06

ACESSE A API E O DASHBOARD EMQX

API REST      → http://localhost:8000
Swagger UI    → http://localhost:8000/docs
EMQX Dashboard → http://localhost:18083  (admin / public)
